From 1990 to 2025: My Journey with Nixpacks

August 21, 2025 • Armin Esmaili

Dependency Hell

Picture this: it's the early 90s. You're trying to install two simple programs on your machine. Everything goes smoothly until you realize one requires an old version of a library, and the other stubbornly demands the new one. Suddenly, you're in dependency hell.

This is where the infamous phrase "Works on my machine" was born — a bitter joke among developers everywhere.

The Birth of Nix

In 2003, a PhD student named Eelco Dolstra asked a simple but revolutionary question:

"What if software builds were as predictable as mathematical functions?"

The result? Nix — a purely functional package manager that guarantees reproducible builds, isolated environments, and zero dependency conflicts.

Nix Principles in a Nutshell

  • Pure builds — Output depends only on inputs
  • Unique package paths — PHP 8.2 and 8.3 can coexist peacefully
  • Declarative — You say what, not how
  • Reproducible — Same input = same output, forever

Enter Nixpacks

Heroku made deployment easy. But it wasn't always predictable.

Nixpacks combines the simplicity of PaaS with the reproducibility of Nix — giving you the best of both worlds.

How Nixpacks Works

It analyzes your code, creates a build plan, and produces a standard OCI image that runs anywhere — with zero surprises.

My Experience with Laravel

I deployed a Laravel app with almost zero configuration. No messy Dockerfiles. No dependency mismatches, or late-night debugging sessions. It just... worked.

It felt like running the app locally — but on a real server.

When Should You Choose Nixpacks?

  • You value reproducibility
  • You hate writing Dockerfiles
  • You want fast, reliable deploys
